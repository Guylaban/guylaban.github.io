# Overleaf Skill

Overleaf projects are editable over a git bridge at `https://git.overleaf.com/<project-id>`. This skill clones the project into a scratch directory, lets you edit the `.tex`/`.bib`/asset files with the normal Read/Edit/Write tools, then commits and pushes back to Overleaf.

## Prerequisites (check before doing anything)

The user must provide — ask with `AskUserQuestion` if any are missing:

1. **Project ID** — the 24-char hex-ish string after `/project/` in the URL when the project is open in Overleaf (`https://www.overleaf.com/project/PROJECTID`).
   - A share URL of the form `https://www.overleaf.com/<long-token>#<hash>` is NOT a project ID and cannot be cloned directly. Ask the user to open the project and copy the URL from the address bar, or paste the project ID.
2. **Git authentication token** — generated by the user at https://www.overleaf.com/user/settings → "Git Integration" → "Create Token". This is NOT the Overleaf password.
3. **A plan that includes Git integration** — required on the user's Overleaf account. If `git clone` returns `403` or `Authentication failed`, their plan likely does not include it; tell the user and stop.

Accept the token via:
- `$OVERLEAF_TOKEN` environment variable (preferred), or
- The user pasting it in chat (treat as sensitive — never echo it back, never write it to a tracked file).

## Clone

```bash
: "${OVERLEAF_TOKEN:?set OVERLEAF_TOKEN}"
PROJECT_ID="<id>"
WORKDIR="$(mktemp -d)/overleaf-${PROJECT_ID}"

# Use a credential helper via env so the token never ends up in the remote URL on disk.
git -c credential.helper= \
    -c credential.helper='!f() { echo "username=git"; echo "password=${OVERLEAF_TOKEN}"; }; f' \
    clone "https://git.overleaf.com/${PROJECT_ID}" "$WORKDIR"

cd "$WORKDIR"
```

Rationale for the credential helper: embedding the token in the remote URL (`https://git:TOKEN@git.overleaf.com/...`) persists it inside `.git/config`, which is a leak risk. The inline credential helper answers the prompt once without storing it.

If the helper approach fails in the user's environment, fall back to:
```bash
git clone "https://git:${OVERLEAF_TOKEN}@git.overleaf.com/${PROJECT_ID}" "$WORKDIR"
```
but **immediately rewrite the remote** after clone so the token is not stored:
```bash
git -C "$WORKDIR" remote set-url origin "https://git.overleaf.com/${PROJECT_ID}"
```

## Explore and edit

Inside `$WORKDIR` the project is a normal LaTeX repo. Use `Glob`/`Grep`/`Read` to find content and `Edit`/`Write` to change it. Typical files:
- `main.tex` (entry point — name varies)
- `references.bib`
- `sections/*.tex`
- image assets

## Commit and push

```bash
cd "$WORKDIR"
git add -A
git -c user.name="${GIT_AUTHOR_NAME:-$(git config user.name || echo Claude)}" \
    -c user.email="${GIT_AUTHOR_EMAIL:-$(git config user.email || echo claude@anthropic.com)}" \
    commit -m "<short, descriptive message>"

git -c credential.helper= \
    -c credential.helper='!f() { echo "username=git"; echo "password=${OVERLEAF_TOKEN}"; }; f' \
    push origin master
```

Overleaf's default (and only) branch is `master`. Never try to push to a different branch — it will fail.

If push is rejected with `non-fast-forward` (someone edited the project in the Overleaf web editor in the meantime):
```bash
git -c credential.helper= \
    -c credential.helper='!f() { echo "username=git"; echo "password=${OVERLEAF_TOKEN}"; }; f' \
    pull --rebase origin master
# resolve any conflicts, then re-run the push command above
```

Do NOT force-push — the git bridge rejects it and you risk clobbering web-editor changes anyway.

## Cleanup

When the task is complete, tell the user where the working copy is and offer to delete it:
```bash
rm -rf "$WORKDIR"
```
Always clean up if the user confirms, or if the clone used the fallback (token-in-URL) approach even momentarily.

## Security rules

- NEVER echo, log, or write `$OVERLEAF_TOKEN` to any file.
- NEVER paste the token into a commit message or PR description.
- NEVER commit a `.env` or config file containing the token into the Overleaf project itself.
- If the user pastes the token in chat, treat the message as sensitive; do not repeat it back.
- Prefer the credential-helper form over URL-embedded credentials.

## Limitations

- Git bridge exposes source only — no compiled PDF. To preview, the user must compile in Overleaf.
- Cannot create new Overleaf projects (no API for that via git).
- Cannot use share links directly; need the project ID.
- History rewrites (`push --force`, `rebase` onto remote) are rejected by the bridge.
- Binary assets larger than Overleaf's per-file limit cannot be pushed.

## When to invoke this skill

Invoke automatically when:
- The user gives an `overleaf.com` URL and asks to read, edit, or update content.
- The user says "update my CV/paper on Overleaf", "push this to Overleaf", etc.

Do NOT invoke for:
- General LaTeX questions not involving an Overleaf-hosted project.
- Compiling LaTeX locally (use `latexmk`/`pdflatex` directly instead).
